# CampusFlow 1.0.0

Release date: 2026-07-15

CampusFlow 1.0 is a portfolio release of a local-first course-planning
prototype. It demonstrates constraint validation, graph algorithms, bounded
untrusted-data parsing, accessible interaction design, offline behavior, and a
least-privilege static deployment. It is not an official University of Toronto
calendar or degree audit.

## Highlights

- Evaluate nested `all` and `any` prerequisite rules without flattening their
  meaning.
- Detect catalog cycles, unknown courses, duplicates, and same-term
  prerequisites.
- Explain conflicts with deterministic minimum-edit repair suggestions under a
  documented cost model.
- Evaluate compositional course and credit requirements against an explicitly
  illustrative program fixture.
- Reorder terms with Pointer Events or equivalent keyboard-accessible controls
  while preserving focus, announcements, persistence, and validation.
- Import and export versioned plans through a bounded, transactional trust
  boundary.
- Validate source-attributed catalog snapshots and explain semantic changes
  between versions.
- Install a content-versioned PWA shell with explicit update activation and
  conservative same-origin caching.
- Deploy one verified artifact through GitHub Pages and smoke-test the published
  document, manifest, worker, and referenced shell assets.

## Verification evidence

The release workflow runs linting, strict type-checking, 52 domain tests, 15
build and deployment script tests, a production build, deterministic raw and
gzip asset budgets, a high/critical dependency audit, Chromium end-to-end tests,
and axe rules tagged WCAG 2.0/2.1 A and AA. The successful Pages run is
[29365490857](https://github.com/nguyengiaphu0304-rgb/campusflow/actions/runs/29365490857).

The release checklist distinguishes this automated evidence from manual checks.
Passing axe rules is not WCAG certification, and a successful smoke test is not
an uptime guarantee.

## Data and privacy

- Course and program data are hand-authored fixtures with explicit provenance;
  they are not scraped or endorsed U of T records.
- Plans remain in browser local storage unless a user explicitly exports a JSON
  backup.
- The application has no account, analytics, backend, or network persistence.
- Exported files are readable JSON and are not encrypted after leaving the
  browser.

## Known limitations

- The fixture omits real enrollment capacity, exclusions, co-requisites,
  transfer credit, calendar-year policy, and timetable conflicts.
- Credit groups are evaluated independently and do not optimize global course
  allocation.
- Automated accessibility checks do not cover a manual screen-reader session,
  browser zoom and reflow review, forced-colors review, or assistive pointer
  devices.
- Browser automation currently targets desktop Chromium; Firefox, Safari,
  mobile browsers, touch, and pen remain unverified.
- Offline use requires one successful online visit, and clearing site data
  removes both the cached shell and local plan.
- GitHub Pages serves the CSP through HTML meta rather than a project-defined
  response header.

## Reproduce the release checks

```bash
npm ci
npm run lint
npm run typecheck
npm test
CAMPUSFLOW_BASE_PATH=/campusflow/ npm run build
npm run budget
CAMPUSFLOW_BASE_PATH=/campusflow/ npm run check:deployment
npm run audit
npx playwright install chromium
npm run test:e2e
```

Exact browser output depends on the installed Chromium build and operating
system packages. CI is the release authority for the committed workflow.
